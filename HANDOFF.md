## ★ 生成流程 SOP(强制,每次必走 —— 用户多次明确要求)

> ⚠️ **同步规则**:本节(SOP)+ 熔断 + 死规则若有任何调整,**必须三处同步**:① `Claude 协作工作流程规范.md` 正文 ② 该文件第 5 章「一页速记」 ③ 本 HANDOFF 对应段落。三者任一不一致即视为改动未完成(对外正式版给同事复用,务必一致)。**另:每次改动 `规范.md` 还须刷新其顶部「版本:vN ｜ 更新时间」(版本号 +1,时间读真实值)。**

> 目标:把「找资料」和「验证试错」的浪费压到 0。**每次开工前先报流程清单,等用户点头再动手。**

### 问题分类(triage,最先判 —— 决定要不要走 SOP)
- **A 纯问答 / 解释 / 闲聊**(不动文件)→ 直接快答,不走 SOP、不报清单、不求口令
- **B 改文件 / 代码 / 数据** → 走下面完整 SOP
- **C 分不清** → 先一句话问「这是要我直接答,还是走规范流程?」

### 第 0 步 · 先给方案分级(决定后面每步的做法)
拿到任务**先判级**,在报清单时写明「这是 L几」(用户可当场纠正):

| 级别 | 典型动作 | 报清单 | 找资料 | 验证 |
|---|---|---|---|---|
| **L1 微改** 纯文案/颜色/字号/间距 | 改标题、换词、调色 | **一句话**报「改 X 文件某处文字」 | grep 1 次定位 | **不自检**,直接交付 |
| **L2 局部逻辑** 单模块内交互/数据/字段 | 改弹窗逻辑、加筛选、改计算 | 报全 5 项清单 | grep + 只读该模块 | **跑 1 次** eval/截图 |
| **L3 流程/跨文件/新功能** 动路由/导航/新模块/多文件联动 | 加页面、改 tab、跨模块联动 | 报全 5 项 **+ 列出所有涉及文件 + 影响面**(路由/全局作用域/三处 pill) | 读相关多文件 + index.html 引用段 | **跑 1 次** + 收尾 `fork_verifier_agent` |

判级原则:**拿不准就低不就高,宁可多报** —— L1 当 L2 报无妨,绝不 L3 当 L1 漏掉影响面。**熔断规则对三级都生效。**

### 第 1 步 · 开工前必报「流程清单」等确认(不可跳过)
拿到任务,**先别动手**,用一段话列清:
- **要读什么** —— 哪几个文件 / grep 什么关键字(精确到文件 + 函数,不写「读一下相关文件」这种模糊话)
- **改哪里** —— 哪个文件的哪个函数 / 哪段
- **做什么** —— 具体改动内容
- **要不要验证** —— 验 / 不验,验的话跑哪 1 次

然后**自排查并写出**:「本次**不做**哪些非必要步骤」(例:不读全量 index.html / app.jsx;纯文案不自检……),证明没有多余动作。
→ 等用户确认方向,再开工。**任何任务都先报、都等确认,包括小改。**

### 第 2 步 · 找资料(0 浪费)
- **永远 grep 定位,不靠记忆猜文件**(历史教训:猜错文件返工)。
- **只读命中的那一个模块** + 它在 index.html 的那行 `<script>`;无关文件一律不读。
- **绝不全量预读** index.html / app.jsx / prd.jsx / version.jsx —— 只在动路由/导航/PRD/版本时读对应那一段。
- 读过的文件记住,不重复读。

### 第 3 步 · 改(优先局部)
- 优先 `str_replace_edit` 局部改;大改才 `write_file`。
- 复制别的模块来改 → **先给要改的顶层标识符加唯一前缀**(全局作用域 last-wins,见第 4·E 节)。
- **不改路由格式**(第 4·C 节)。

### 第 4 步 · 验证(最多 1 次,其余交后台)
- **纯文案 / 纯样式改 → 不自检**,改完直接交付(静态文字/CSS 不可能崩)。
- **涉及 JS 逻辑 / 数据 / 交互 → 只跑 1 次关键验证**:**优先用 eval 查 DOM / 数据一次确认**;截图只为「给人看视觉」、易失败,不连试(见死规则③)。
- 代理后台验证 → **固定快捷登入脚本**(切 tab→找 AC範例6→点「填入」→点登入→跳页),一次脚本跑完(见第 7 节)。
- 剩余检查交 `fork_verifier_agent`,不阻塞等它。

### 第 5 步 · 收尾(不等提醒)
- `modules/version.jsx` 顶部加一条新版本记录(上一条 `current` 改 `false`)+ **同步在 `CHANGELOG.md` 顶部加对应条目**(含日期+时间)。
- 回答简短,先结果后理由。

### 第 6 步 · 收尾自查表(死规则⑧,B 类强制输出)
改完收尾**必须输出一份自查表**,逐条 ✓/✗:★全程无不诚实 / ①报清单等口令 / ②无 scrollIntoView / ③验证≤1次 / ④四步没压缩+复述id / ⑤确认有 a/b / ⑥计时如实 / ⑦无自创格式 / ⑧该给的输出项(报清单·计时·改动结果·version·CHANGELOG·本表)全到位。**任一 ✗(尤其 ★)→ 当场走违规三步:① 承认 → ② 根因 → ③ 修复方案,禁止只停在承认。**

## ★★ 防卡死「熔断」规则(强制,优先级最高 —— 用户明确要求)

> 历史最大时间黑洞 = 闷头反复搜/改/验。用「尝试次数」当时间代理,超限**立刻停**,转替补流程,绝不硬试到超时。

### 触发条件(任一即熔断)
- **找资料**:grep / 读文件累计 **3 次**仍没定位到要改的地方
- **改同一处**:改→报错→再改,**反复 2 次**仍没通过
- **验证**:同一验证(eval / 登入脚本 / 截图)**失败 2 次**

### 超限后(立刻做)
1. **停手** —— 不再重复搜/改/验。
2. **一句话报现状** —— 卡在哪、试了什么、为什么没成。
3. **给用户 2 个选项**拍板:
   - (A) 带已知信息**换方向**继续(说清新方向)
   - (B) 用户**补一个线索**(关键字 / 文件名 / 截图),我直接定位。
4. **绝不闷头硬试到超时**。

## ★★★ 死规则(强制 —— 用户复盘后要求写入,不可重犯)

> 以下 7 条是真实犯过的错(本项目复盘累积)。每条都给「死规则」,动手前过一遍。
> **适用范围**:违规①~⑥ **只对 B 类(改文件/代码)生效**,A 类纯问答豁免;**⑦(禁自创回复格式)A+B 全类通用;⑧(收尾自查)B 类必出表、A 类仅守 ★ 诚实项**。
> **元规则(防冲突)**:新增/改动死规则时,必须声明其**适用范围(A/B/C 哪类)**,并检查与现有规则是否冲突;冲突先解决再写。

### 违规①「明确需求 ≠ 已确认」仍跳过报清单(最严重)
- **犯错**:用户给了清晰 mockup,我把「需求明确」当成「已确认」,直接 grep+改代码,跳过「先报清单等确认」。
- **死规则**:**任何任务,哪怕用户给了 mockup / 截图 / 精确描述,执行前一律先报流程清单,等用户点头才动手。** 需求明确只是省了「问需求」,不省「报清单确认」这一步。

### 违规② 用了被禁的 scrollIntoView
- **犯错**:截图驱动时图省事用 `scrollIntoView`,把抽屉弄关、白费一次截图(系统规则明确禁止)。
- **死规则**:**滚动只用 `el.scrollTop = …` / `scrollTo`,永不用 `scrollIntoView`。** 动 DOM 滚动前先想一遍禁令。

### 违规③ 截图反复试错、超熔断阈值
- **犯错**:截图试了 5~6 次才搞定,而 eval 早就确认 DOM 正确,却继续纠结截图视觉。
- **死规则**:**验证优先用 eval 查 DOM,一次确认就停。** 截图只为「给用户看视觉」,不是「自己验证功能」;截图失败 **1 次**就改用 eval 确认 + 交 verifier,绝不连试。
- **附:React 抽屉态截图坑** —— `save_screenshot` 会重载页面、清掉 React state(抽屉/弹窗会关)。要截 React 临时态:把「跳页→点开→滚动」全写进**同一个 save_screenshot 的 steps**里跑(别先 eval 开再截,中间会被重载清掉)。

### 违规④ 执行确认 —— 两种确认模式,选方案与确认执行永不同句
- **根因**:把「确认执行?(回 a/b)」写成一句,用户回「a」时分不清"选方案"还是"下口令",擅自就跑(反复犯)。
- **铁律**:**「选方案的 a/b/c」与「确认执行」永不同句**。动文件前必须先有一句专门的执行确认通过。
- **四步次序(绝不压缩成一条消息)**:① 我只给**方向选项** a/b/c(不提口令)→ ② 用户选 → ③ 我**根据所选方向报「具体怎么做」的清单** → ④ **这时才**问「确认执行?」等口令。根因:曾把 ①④ 塞进同句(「选 a…回口令我就动手」),让用户一边选方向一边被催口令——明令禁止。下面两种确认模式择一:
  - **执行确认只认口令白名单**(大小写不限):`执行 / 做 / go / 确认 / 可 / 可以 / 行 / OK / K`。报完清单问「确认执行?」,使用者主动打口令才动手。
  - **绝不在回复里写 a/b/c、q/w/t 这种单字母执行代号**(根因:只要写了代号,系统就把它回显成 usera/usrq,被误当确认→擅自执行;换哪个字母都没用,要从根上不写代号)。
  - **选方案仍可用 a/b/c**(只表意图、不触发执行):选完后仍需用户打口令才动手。
  - **兜底口令**(=执行,大小写不限):`执行 / 做 / go / 确认 / 可 / 可以 / 行 / OK / K`。
  - 确认项之外的回应(好/不错)不触发执行;回的是确认项外内容 → 反问「先讨论新话题还是继续执行?」
  - **放行前核验有效确认信号 + 复述 id**:空消息/纯标点/系统占位/非确认内容一律不执行;动文件前先写一行「据以执行:你 mXXXX 的 q」引用真实 id,引用不出就停(曾把系统占位的 usera 误当确认擅自执行)

### 违规⑤ 向用户确认时用 a/b/c 选项
- **死规则**:凡需用户拍板,选项列成 `a / b / c…` 便于快答(如 a保留/b调整/c回退)。用户可不选直接说需求,或「选项+补充」。
- **含二选一也必须给选项**:哪怕只是「要 / 不要、是 / 否、改 / 不改」,也写成 `a 要 / b 不要` 这种带字母的选项,**绝不用开放式问句**(如「要不要…?」「你觉得呢?」)甩给用户回字。根因:「凡需拍板」曾被择窄解读成「仅多方案才用 a/b/c」,把二选一漏掉——明确堵死。

### 违规⑥ 执行分段计时,完成后回报各阶段耗时
- **死规则**:收到执行口令后,**第一动作 = `show_html` 加载预览(index.html)→ 再 `eval_js → Date.now()` 读时钟打点**(没这两下不准碰文件);每个大阶段切换再读一次;完成后回报总耗时 + 各阶段(① 找资料 ② 改 ③ 验证 ④ 收尾)耗时。L1 微改只报总耗时,L2/L3 分段报。
- **🚫 禁编造耗时 + 禁用「未加载」当懒借口**:耗时只能由两次真实时钟读数相减得来。**「无计时」不是默认退路 —— 只有在 `show_html` 加载后时钟仍读不到时才准写「无计时」,且该情况自查表 ⑥ 记 ✗/异常,绝不记 ✓。裸跑一下 `eval_js` 撞「viewport not ready」就写「无计时」是偷懒造假(真犯过:多轮未先 show_html 就报「预览未加载」);编个「约 X 分钟」同样是造假(真犯过:v3.7.55 那轮「约 4 分钟」)。
- **原理**:时间来自预览页 JS 时钟(与订单时间戳同源),精确到毫秒;计时区间=「确认口令→执行完成」,不含用户思考/打字。**>60 秒带分钟单位(如 5 分 47 秒),≤60 秒只写秒。**
- **分段报告格式 + 逐点打点纪律**:
  ```
  ⏱ 本次执行耗时 N 分 M 秒
  起始时间 HH:MM:SS
    1. 找资料  t1-t0
    2. 改      t2-t1
    3. 验证    t3-t2
    4. 收尾    t4-t3
    5. 检查    末-t4
  结束时间 HH:MM:SS
  ```
  **起始时间**=确认口令刻(t0,加载预览后首次打点);**结束时间**=整个流程结束、**含输出自查表之后**那一刻(因「检查」也计入耗时,故自查表写完才读末时钟、再补计时块)。起止时间用真时钟格式化(`toLocaleTimeString('zh-CN',{hour12:false})`)不估。分段耗时**必须靠真打点**:每个阶段切换点各读一次。**绝不能只打首尾两点再用「纯文档未拆段」之类跳过分段**(真犯过:只打 t0/t1 就报总数 158s)。**每行必须真换行**:用 Markdown 列表/空行,使起始时间、1~5、结束时间渲染后各占一行,绝不用全角空格连成一大条(真犯过)。L1 微改可只报总时;L2/L3 必分段。

### 违规⑦ 自创回复编号前缀 / 装饰格式(适用 A+B 全类)
- **犯错**:在回复开头自创「回覆 mXXXX ｜ 本条 #G3」这种编号前缀 —— 规范从没这条,是脑补加的,既多余又和死规则④「回复里绝不写可被系统回显的代号」冲突。
- **死规则**:**回复只按规范要素写(报清单 / 口令白名单 / 计时 / a·b·c 选项等),不自创任何编号前缀、序号标签、装饰格式。** 想加新格式先问用户、写进规范再用。

### 违规⑧ 收尾自查清单 + 违规三步(适用 A+B 全类)
- **犯错**:整轮反复犯死规则(编造耗时、漏选项、压缩四步、自创编号),且几乎都靠用户点出才发现 —— 缺「收尾主动自查」这道闸门;承认了却常停在承认、不给修复。
- **死规则**:**每个 B 类任务收尾,强制输出一份自查表**,逐条 ✓/✗:
  | # | 检查项 |
  |---|---|
  | ★ | **全程无不诚实**(没编数字/没编借口/没含糊蒙混,做不到就如实说) |
  | ① | 报清单+等口令才动手(没抢跑) |
  | ② | 没用 scrollIntoView |
  | ③ | 验证 ≤1 次 / 纯文案免验 |
  | ④ | 四步次序没压缩、口令复述了 id |
  | ⑤ | 给用户确认时有 a/b 选项(含二选一) |
  | ⑥ | 计时如实(读到就报、读不到写无计时) |
  | ⑦ | 没自创编号前缀/装饰格式 |
  | ⑧ | 该给的输出项全到位(报清单/计时回报/改动结果/version.jsx/CHANGELOG/本自查表,逐项核有没有漏) |
- **任何一项 ✗(尤其 ★)→ 当场走违规三步**:① 承认违了哪条 → ② 找根因 → ③ 给修复方案(能落文件就落),**禁止只停在承认**。
- **适用**:A 类纯问答收尾不强制出表(但 ★ 诚实项任何时候都成立);B 类必出。

## 4. 强制规则(易忽略)

### A. 版本记录(强制,别等提醒)
每次改代码 → `modules/version.jsx` 顶部加一条 + **`CHANGELOG.md` 顶部同步加一条**:
- 新条 `current:true`,把上一条改 `current:false`(永远只有一条 current)
- **`ver` 编号规则**(`vX.Y.Z`,进位由改动规模决定,拿不准就只 +Z):
  - **Z 修** = 默认每次 +0.0.1(bug / 文案 / 样式 / 单点小逻辑)
  - **Y 次** = 攒够一个完整功能模块做完 → +0.1.0,Z 归零(v3.7.x → v3.8.0)
  - **X 主** = 大版本里程碑 / 架构级重构 → +1.0.0,Y、Z 归零
- CHANGELOG 条目含 版本号 + 日期 + 时间(用页面 `Date.now()` 读),标签中文:修复/新增/修改/删除/功能
- `changes:[{ type:'fix'|'add'|'modify'|'remove'|'feat', text:'...' }]`
- 小改动(bug / 文案 / 样式)也要记,写清改了什么 + 为什么

### B. 导出代码(强制排除)
排除 `uploads/`、任何 `舊備份*/old*/backup*/_bak*`、`screenshots/`。
做法:`run_script` copy 需要的档案到临时 `_export/` → `present_fs_item_for_download` 打包 zip → 完成后删 `_export/`。**不要**直接打包整个根目录(会带出 uploads)。

### C. 不要改路由格式
`home` / `section:<名>` / `mod:<navKey>` / `prd_overview` / `prd_home` / `version` / `phase:P0` / `feat:P0-1`。一改全断。parseRoute(app.jsx ~138 行)决定每个路由的 parent(返回按钮按层级回退)。

### D. 优先级 pill 三处渲染
首页 hc-row / 大项页 fc-child / PRD 卡片 —— 改样式三处同步。

### E. ⚠️ 各 jsx 模块「共享全局作用域」(易踩坑)
所有 `<script type=text/babel>` 模块的**顶层声明会泄漏到全局**(Babel 把 `const`→`var`、`function` 挂 window),按 index.html 的 `<script>` 顺序**后载入覆盖先载入**(last-wins)。
- 后果:两个文件若有同名顶层 helper(`WRCard`/`DocRow`/`T`/`fmtDT`...),后载入的会覆盖先载入的。
- 现状:`modules/withdraw_review.jsx`(line 31,商户「代理提款审核」)是 `modules/my_withdraw_progress.jsx`(line 50,代理端,**更晚载入**)的副本。两边相同的 helper 没事;但 withdraw_review 里**任何被改动 / 与代理端分歧的顶层函数,必须改唯一名**,否则被代理端旧版覆盖、改动失效。
- 已处理:withdraw_review.jsx 的 doc 卡片 helper 已改 `WRCard/FSCard/POCard/DocRow → WRV_WRCard/WRV_FSCard/WRV_POCard/WRV_DocRow`;翻译函数 `T → WRV_T`(且强制读中文字典,见下条)。后续再改直接编辑 `WRV_*`。
- ⚠️ **语言泄漏**:withdraw_review 是商户后台,必须**永远中文**。它曾用 `const T = window.t` 跟随代理端 `useAgentLang`,代理后台切英文时商户这页也变英文。已修:`WRV_T` 强制读 `window.APS_I18N.zh`,并移除组件内 `useAgentLang()` 订阅。任何商户后台模块复制自代理端的,都要确认不跟随 useAgentLang。
- 通则:复制别的模块来改时,**先把要改的顶层标识符加唯一前缀**再动手。

## 7. 几个有用的 window 全局钩子

- 各模块挂在 window 上(如 `window.FrontendModule`、`window.AgentDashboardModule`),app.jsx 用 `<window.XxxModule/>` 渲染
